# Roadmap: Voice Agent for Local Businesses

## Overview

The project builds a fully autonomous AI voice agent for local businesses in five delivery phases. Phase 1 lays the foundation: the FastAPI backend, database schema, multi-tenant isolation, and Retell webhook verification. Phase 2 adds the RAG knowledge base so callers can ask questions and get grounded answers. Phase 3 implements the full appointment lifecycle — booking, rescheduling, cancellation — wired into Google Calendar. Phase 4 closes the loop with email notifications to callers and business owners, plus callback escalation when the agent cannot help. Phase 5 deploys the system to a public URL and delivers the CLI scripts Russell uses to onboard new clients.

## Phase Details

### Phase 1: Foundation
**Goal**: The backend exists, accepts verified Retell webhooks, stores per-client data in isolation, and encrypts OAuth tokens at rest
**Depends on**: Nothing (first phase)
**Requirements**: VOICE-01, VOICE-02, TENANT-01, TENANT-02, INFRA-02, INFRA-03
**Success Criteria** (what must be TRUE):
  1. A POST to the Retell webhook endpoint with a valid HMAC signature is accepted; one with an invalid signature returns 401
  2. Two clients can be inserted into the database with different client_ids and their data does not cross-pollute in queries
  3. A Google OAuth token written for client A is stored encrypted in the database and cannot be read as plaintext by inspecting the row
  4. The health check endpoint at `/health` returns HTTP 200 with no authentication required
  5. The FastAPI app starts locally and connects to PostgreSQL with pgvector extension available
**Plans**: 3 plans

Plans:

- [x] 01-03-PLAN.md — FastAPI app wiring, Retell HMAC middleware, /health endpoint, router skeleton, auth + health tests

### Phase 2: RAG Knowledge Base
**Goal**: Callers can ask a business question and receive a grounded answer generated from that client's ingested documents, with graceful fallback when no relevant content is found
**Depends on**: Phase 1
**Requirements**: RAG-01, RAG-02, RAG-03, RAG-04
**Success Criteria** (what must be TRUE):
  1. An uploaded PDF, DOCX, TXT, or CSV is chunked, embedded with OpenAI embeddings, and stored in the client's vector table
  2. Re-ingesting the same document removes the old chunks and replaces them without duplicates or requiring a full rebuild
  3. A caller question that matches ingested content returns an answer generated by Claude Haiku grounded in retrieved chunks
  4. A caller question with no matching content above the similarity threshold returns a transfer-to-team signal rather than a hallucinated answer
**Plans**: 3 plans

Plans:
- [x] 02-01-PLAN.md — Document ingestion service: parsers, chunker, OpenAI embedder, DB write/delete, unit tests
- [x] 02-02-PLAN.md — Admin web panel: Jinja2 upload/list/delete UI, admin router, tests
- [x] 02-03-PLAN.md — RAG query service: similarity search, Claude Haiku answer generation, Retell webhook wiring, tests

### Phase 3: Calendar Operations
**Goal**: A caller can book a new appointment, reschedule an existing one, or cancel one — all handled autonomously by the agent against Google Calendar with no human involvement
**Depends on**: Phase 2
**Requirements**: APPT-01, APPT-02, APPT-03, APPT-04, APPT-05, APPT-06
**Success Criteria** (what must be TRUE):
  1. A caller requesting a specific date and time that is free results in a confirmed Google Calendar event with the caller's name, phone, and email captured
  2. A caller requesting a time that is already booked is offered the next 2-3 available slots as alternatives
  3. A caller who declines all alternatives can specify an available window and the agent finds a matching slot within 30 days
  4. A caller who provides their name and appointment date can reschedule to a new time and the original Google Calendar event is updated
  5. A caller who provides their name and appointment date can cancel and the Google Calendar event is removed after the agent confirms intent
**Plans**: 4 plans

Plans:

- [ ] 03-02-PLAN.md — Google Calendar service: OAuth refresh, get_free_slots, create/update/delete events + tests
- [ ] 03-03-PLAN.md — Appointment orchestration service: booking intake, alternatives, reschedule/cancel + tests
- [ ] 03-04-PLAN.md — Retell webhook wiring: 6 calendar tool dispatch branches + tests

### Phase 4: Notifications and Edge Cases
**Goal**: Every appointment action triggers email confirmations to the caller and alerts to the business owner; unresolvable calls route to a callback request rather than failing silently
**Depends on**: Phase 3
**Requirements**: VOICE-03, NOTIF-01, NOTIF-02, NOTIF-03
**Success Criteria** (what must be TRUE):
  1. After a booking, reschedule, or cancellation, an email confirmation is delivered to the caller's email address with the appointment details
  2. After any appointment action, an email notification is delivered to the business owner containing the caller's details and the action taken
  3. When the agent cannot find an available slot or cannot understand the caller after 2 attempts, a callback request email is sent to the business owner
  4. A caller who explicitly requests to speak to someone triggers the callback escalation path rather than the agent continuing to retry

### Phase 5: Admin and Deployment
**Goal**: The system is live on a public HTTPS URL and Russell can onboard a new client, run their OAuth flow, and ingest their knowledge base entirely via CLI scripts
**Depends on**: Phase 4
**Requirements**: TENANT-03, TENANT-04, TENANT-05, INFRA-01
**Success Criteria** (what must be TRUE):
  1. Running the create-client CLI script with a business name and phone number creates a new client record in the database
  2. Running the oauth CLI script for a client opens the Google OAuth consent flow and stores the resulting encrypted tokens
  3. Running the ingest CLI script with a client_id and file path ingests the document into that client's knowledge base
  4. The backend is accessible at a public HTTPS Railway URL and Retell can reach the webhook endpoint from the internet
